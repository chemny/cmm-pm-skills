# cmm-pm-skills · 流水线总览

> 一套**产品经理工作流** skills：从市场到落地的端到端链路。
> 核心纪律统一在根目录 **`CONVENTIONS.md`**（母版），每个插件内置同步副本，安装即继承。改母版后跑 `./sync_conventions.sh`。

## 总协同（站在所有阶段之上）

**`/cmm-pm-skills`** 是套件的总入口/总协同：一句话上手（整套或单块），为每个产品项目建/读台账 `_runs/<项目>/_state.yaml`、显示进度看板、按阶段守门（风险结论→用户拍板）、领航下一步。全套已合并为**单插件 `cmm-pm`**，故总入口可直接驱动同插件内各阶段。它补上了"换会话后项目状态会丢"的洞。

## 全链路（① → ⑪）

| 阶段 | 命令 | 输入 → 输出 | 归属 |
|---|---|---|---|
| ① 市场分析 | `/market-analysis` | 需求 → 市场/用户/买单人/竞品/颠覆机会（数据定） | PM |
| ② 需求洞察 | `/insight` | ①结论 → 真需求(JTBD)+风险结论+一手验证计划（默认"强假设"） | PM |
| ③ 产品战略 | `/strategy` | ①② → 定位/壁垒/商业模式/定价/打法 | PM |
| ④ PRD | `/write-prd` | ①②③ → 完整可开发 PRD（需求+功能+流程+交互+数据+边界） | PM |
| ⑤ 界面设计 | `/wireframe` | ④ → Mermaid流/态 + 可点击 HTML 原型 + 文字规格 | PM |
| ⑥ 技术方案 | `/tech-design` | ④⑤⑦ → 选型/架构/数据/**解 spike**/冲突检查 | ⚙️ 工程(提案) |
| ⑦ 用户故事 | `/write-stories` | ④⑤ → 故事+验收(可追溯) | PM |
| ⑧ 测试 | `/test-scenarios` | ⑦ → 测试用例(happy/边界/错误) | PM |
| ⑨ 研发 | —（外部/Superpowers） | ⑥⑦ → 代码 | ⚙️ 工程·不在本套件 |
| ⑩ 上线 | `/plan-launch` `/battlecard` `/growth-strategy` | → GTM | PM |
| ⑪ 数据/迭代 | `/analyze-cohorts` `/north-star` `/analyze-test` | → 指标 → 回到 ② | PM |

> 边界：⑥技术方案与⑨研发本属工程；⑥在本套件以"技术提案"形态存在（解阻塞、到可估算为止），最终架构由工程定。

## 统一纪律（CONVENTIONS.md 摘要）
- **§0 真实性>完整性**；**§0a 你是辅助者、用户是决策者**；**§0b 不替用户改需求(pivot 是用户决定)**；**§0c 协作+确认核心要素+定了就全力帮**。
- §1 提问(ClassA问/B查·说人话·不逼选)｜§2 原子词整词搜｜§3 竞品按品类分｜§4 研究提纲先行+三角配额+真搜索引擎+信源分级｜§5 反浅显(薄了去搜真东西)｜§6 风险+机会两面·不因巨头否定｜§7 方法论后台·人话标题｜§8 血缘自检+回退补搜+多角色红队(含**卖点守护**)｜§9 禁止编造｜§10 画像按人群细分。

## 怎么用
- 按阶段顺序触发命令；每个命令继承上游、不重造；产出存 markdown（图另出可视化 HTML）。
- 每个阶段结尾给"风险结论"，**做不做由用户决定**。

## 维护
- 改纪律 → 只编辑根 `CONVENTIONS.md` → 跑 `./sync_conventions.sh` 同步 9 插件。
- 改命令/技能 → 跑 `python3 validate_plugins.py` 校验契约。
- 设计蓝图见 `_design/`；端到端示例见 `_runs/voice-ime/`（sayit 产品全链路）。
